## Public API

The public API of this library is **everything exported from the main entry point** (e.g. `src/index.ts`). All other files are internal implementation details.

- Do not add exports to `src/index.ts` without discussion — every export becomes a public commitment.
- Do not remove or rename existing exports without deprecating them first (see [Deprecation Policy](#deprecation-policy)).
- Do not change the signature of an exported function in a breaking way within a major version.
- Document all exported symbols with JSDoc/docstrings.

## Versioning

This library follows [Semantic Versioning](https://semver.org/):

| Change type                              | Version bump |
| ---------------------------------------- | ------------ |
| Backwards-compatible new feature         | Minor        |
| Backwards-compatible bug fix             | Patch        |
| Breaking change (removed/renamed export) | Major        |

## Deprecation Policy

Before removing or changing an exported symbol:

1. Mark it as deprecated in the current release with a `@deprecated` JSDoc tag.
2. Document the replacement in the deprecation message.
3. Keep the deprecated symbol for at least one minor version.
4. Remove it only in the next major version.

## Adding a New Feature

1. Add the implementation under `src/`.
2. Export it from `src/index.ts` if it should be public.
3. Add tests in `tests/`.
4. Add an example in `examples/` if the feature is non-trivial.
5. Document the new export in [docs/api/README.md](docs/api/README.md).

## What to Avoid

- Do not add runtime dependencies without careful consideration — every dependency increases the bundle size for consumers.
- Do not mutate inputs passed to library functions — always return new values.
- Do not use `console.log` in library code — it pollutes the consumer's output.
- Do not make breaking changes without a major version bump.
- Do not commit build artifacts (`dist/`, `build/`) — these are generated by CI.
